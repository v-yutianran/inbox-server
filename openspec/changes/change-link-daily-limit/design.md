## Context

worker 使用同一个 `consume` 循环处理多类队列，`QueueLimits` 同时携带固定窗口、日限额和消费间隔。当前 link 配置为 120 条/6 小时与 480 条/日；此外 server、worker 容器未设置业务时区，日计数实际按 UTC 日期生成 Redis key。

## Goals / Non-Goals

**Goals:**

- link 只受上海自然日 500 条上限和既有 5 秒消费间隔约束。
- text、file、article 继续使用既有固定窗口与日限额逻辑。
- 部署切换当天不因时区换桶而放大可用配额。

**Non-Goals:**

- 不删除通用固定窗口限速能力。
- 不修改下游 Cubox 分发、重试、去重或 DLQ 行为。
- 不清理历史固定窗口 Redis key。

## Decisions

1. 使用现有整数配置的 `window_count=0` 表示禁用固定窗口。消费器仅在该值大于 0 时请求固定窗口令牌。这样不改变共享 `QueueLimits` 数据结构，避免影响其 20 个直接引用；相比把字段改为可空类型，改动面更小。
2. link 的 `daily_limit` 设为 500，消费间隔保持 5 秒。日限额仍在出队前检查，达到上限时不取出待处理项。
3. server、worker 显式配置 `TZ=Asia/Shanghai`，使现有基于本地日期的日计数 key 与业务自然日一致。
4. 切换当天将切换前记录的旧 UTC link 日计数与新上海日 key 已有计数相加后写回。UTC 换日会把同一个上海自然日拆到两个 key，取较大值会漏算其中一段；迁移仅执行一次并保留预检证据，避免重复部署再次累加。迁移只影响计数，不移动队列项。

## Risks / Trade-offs

- [风险] `window_count=0` 成为约定值，未来调用者可能误解 → 在消费器判断、配置注释和行为测试中固定该语义。
- [风险] 部署期间 worker 继续增加旧计数，造成迁移竞态 → 先停止 worker，再读取并迁移计数，随后重建启动。
- [风险] 继承旧 UTC 计数可能包含上海自然日前一晚的少量发送 → 采用保守计数，优先保证切换日不突破 500。

## Migration Plan

1. 停止 worker，记录旧 UTC 日 key、新上海日 key及其计数。
2. 将旧 UTC 日计数与新上海日 key 已有计数相加后写入新 key，并设置覆盖下一个上海自然日边界的过期时间。
3. 构建并强制重建 server、worker，验证容器时区、健康状态和队列消费。
4. 若需回滚，恢复旧镜像与 Compose 配置；保留新日计数 key不会破坏旧 UTC 逻辑。
